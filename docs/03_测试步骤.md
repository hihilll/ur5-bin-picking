# 详细测试步骤（在 Ubuntu 上执行）

> 原则：**自底向上、逐个隔离**。先单独验证每个硬件/节点，再两两联调，最后整机。
> 任何一步不过就停下排查，不要急着往下走。

## ⚠️ 安全须知（涉及真实机械臂）
- 测试机械臂运动前：**手握急停**，周围无人无障碍，**第一次把速度调到最低**。
- UR 示教器先在**手动/低速模式**下试，确认轨迹安全再切自动。
- 夹爪通电前确认接线正确，先用 `simulate:=true` 走通逻辑再上真夹爪。

## 测试 0：编译验证（不接硬件）

```bash
cd ~/ros2_ws
rosdep install --from-paths src --ignore-src -r -y
colcon build --symlink-install
source install/setup.bash
```
**预期**：5 个包全部 `Finished`，无 error。
**判定**：
```bash
ros2 pkg list | grep bin_picking      # 应列出 5 个包
ros2 interface show bin_picking_interfaces/msg/ObjectPose   # 能打印字段
```
**常见问题**：
- `bin_picking_interfaces` 没编出来 → 先单独 `colcon build --packages-select bin_picking_interfaces` 再 source。
- Python 节点找不到 open3d → `pip install open3d opencv-contrib-python pymodbus pyserial`。

## 测试 1：相机单独验证（Gemini2）

```bash
ros2 launch orbbec_camera gemini2.launch.py
```
新开终端：
```bash
ros2 topic list | grep -E "points|image|depth"
ros2 topic hz /camera/depth_registered/points     # 话题名以实际为准
```
**预期**：点云话题稳定输出（如 ~15-30 Hz）。
**可视化**：
```bash
rviz2
# Fixed Frame 设为 camera_link / camera_depth_optical_frame
# Add -> PointCloud2 -> Topic 选点云话题
```
**判定**：RViz2 中能看到料框和零件的 3D 点云。
**常见问题**：
- 无设备权限 → 重跑 `install_udev_rules.sh` 并重插 USB。
- 话题名和我代码里的 `input_topic` 不一致 → 记下真实名，后面改 `perception_params.yaml`。

## 测试 2：UR5 连接单独验证

```bash
# 上位机
ros2 launch ur_robot_driver ur_control.launch.py \
    ur_type:=ur5 robot_ip:=192.168.1.102 launch_rviz:=true
```
在 UR 示教器运行 **External Control** 程序。
**判定**：
```bash
ros2 topic echo /joint_states --once        # 能读到 6 个关节角
```
- RViz2 里模型姿态与真实机械臂一致；手动拖动真实臂，RViz2 跟随。
**常见问题**：
- 连不上 → `ping 192.168.1.102` 通不通；IP/网段、External Control 里上位机 IP 是否填对。
- IP 不是 102 → 用真实 IP，并记下来后面 launch 用。

## 测试 3：夹爪单独验证（先模拟，后真机）

**3a. 模拟模式（不接硬件，验证逻辑）**
```bash
ros2 run bin_picking_grasp gripper_driver --ros-args -p simulate:=true
```
另开终端：
```bash
ros2 service call /gripper/set_gripper bin_picking_interfaces/srv/SetGripper \
    "{width: 0.05, force: 50.0, speed: 50.0}"
```
**预期**：返回 `success: true`，节点日志打印 `[模拟] 写寄存器 ...`。

**3b. 真机模式**
```bash
ls /dev/ttyUSB*                              # 确认夹爪串口号
ros2 run bin_picking_grasp gripper_driver --ros-args \
    -p simulate:=false -p port:=/dev/ttyUSB0
```
> ⚠️ 真机前提：已按沃姆协议手册填好 `gripper_driver.py` 的 `REG_*` 和 `POS_COUNT_MAX`。
```bash
# 全开
ros2 service call /gripper/set_gripper bin_picking_interfaces/srv/SetGripper "{width: 0.05, force: 30.0, speed: 30.0}"
# 全闭
ros2 service call /gripper/set_gripper bin_picking_interfaces/srv/SetGripper "{width: 0.0, force: 30.0, speed: 30.0}"
```
**判定**：夹爪实际开合到位。
**常见问题**：
- 串口打不开 → `sudo usermod -aG dialout $USER` 后重新登录；确认 `port` 正确。
- 动作方向/行程反了 → 调 `width_to_counts` 映射或 `POS_COUNT_MAX`。

## 测试 4：手眼标定验证（阶段1）

用 easy_handeye2 完成 Eye-to-Hand 标定后，把结果填入
`bin_picking_bringup/config/handeye_result.yaml`，重新 `colcon build`。

**发布并检查 TF：**
```bash
ros2 run tf2_ros static_transform_publisher ...   # 由 system.launch 自动起
ros2 run tf2_tools view_frames                    # 生成 frames.pdf 看坐标树连通
ros2 run tf2_ros tf2_echo base_link camera_link   # 打印外参，与标定值一致
```
**精度验证（关键）**：在相机视野放一个已知点（如标定板角点），
用感知/手点出它在相机系坐标 → 经 TF 转到基座系 → 让 UR5 移动过去，
**误差应在毫米级**。误差大说明标定不准，需重标。

## 测试 5：感知节点验证（阶段2）

把零件 CAD 放到 `bin_picking_description/meshes/part.stl`。
```bash
# 终端1：相机
ros2 launch orbbec_camera gemini2.launch.py
# 终端2：感知
ros2 launch bin_picking_perception perception.launch.py \
    cad_model_path:=$(ros2 pkg prefix bin_picking_description)/share/bin_picking_description/meshes/part.stl
# 终端3：看输出
ros2 topic echo /detected_objects
```
**预期**：料框里放几个零件，`/detected_objects` 输出位姿，`fitness` 越接近 1 越好。
**可视化**：RViz2 加 PointCloud2 + MarkerArray(`/detected_markers`) + TF(`object_0` ...)，
绿色球应贴在零件上，TF 坐标系朝向合理。

**调参顺序（按需）** —— 改 `perception_params.yaml` 后重启感知节点：
1. `workspace_min/max`：先把工作区裁到只剩料框内（否则背景干扰）。
2. `voxel_size`：约为零件最小特征的 1/5~1/10；太大丢细节，太小慢。
3. `cluster_eps`/`cluster_min_points`：让每个零件被分成独立簇。
4. `min_fitness`：先调低(0.2)看有没有输出，再逐步提高过滤误检。

**判定**：单个/少量堆叠零件能稳定输出正确位姿（CAD 在 RViz 里贴合点云）。
**常见问题**：
- 完全没输出 → 多半是 `workspace_*` 把零件裁掉了，或 `input_topic` 名不对。
- 位姿乱跳 → 零件对称导致歧义，或 voxel 太大；考虑给零件加非对称特征。

## 测试 6：抓取规划验证（不动机器人）

```bash
# 在测试5基础上，再起规划（需要 base_link<-camera 的 TF，即手眼已发布）
ros2 run bin_picking_grasp grasp_planner --ros-args \
    --params-file $(ros2 pkg prefix bin_picking_grasp)/share/bin_picking_grasp/config/grasp_params.yaml
ros2 topic echo /grasp_candidates
```
**预期**：输出基座系下的抓取位姿列表，已按 score 排序。
**可视化**：RViz2 看 `/grasp_markers`，**绿色箭头=最佳抓取**，其余红色。
箭头应指向零件、姿态适合夹爪从上方接近。
**判定**：最佳抓取箭头位置/朝向合理。
**常见问题**：
- 无输出且日志报 `TF 查询失败` → 手眼 TF 没发布（先跑 system.launch 或手动发静态 TF）。
- 抓取朝向不对 → 调 `grasp_annotations`（CAD 系下的抓取位姿）。

## 测试 7：MoveIt 运动验证（先空间，后真机干跑）

**7a. 纯 MoveIt 规划（RViz 拖动，不接真臂）**
```bash
ros2 launch ur_moveit_config ur_moveit.launch.py ur_type:=ur5 launch_rviz:=true
# RViz MotionPlanning 面板：拖动目标 -> Plan -> 看轨迹动画。确认规划组名 = ur_manipulator
```

**7b. 执行器模拟模式（不动真臂，只打印）**
```bash
# 不装 moveit_py 或无真臂时，executor 自动进模拟模式
ros2 run bin_picking_grasp grasp_executor
ros2 service call /pick_place/run std_srvs/srv/Trigger {}
```
**预期**：日志依次打印 开夹爪→预抓取→抓取→闭合→抬起→放置… 每步成功。
**判定**：整个取放**逻辑序列**走通。

---

## 测试 8：抓取链联调（真臂，低速干跑）

> ⚠️ 手握急停，速度调到最低，先不放零件空跑轨迹。
```bash
ros2 launch bin_picking_bringup system.launch.py \
    robot_ip:=192.168.1.102 \
    cad_model_path:=/abs/.../meshes/part.stl \
    enable_grasp:=true
# 夹爪没接好就先: 在 grasp_params.yaml 设 gripper_driver.simulate: true
```
触发：
```bash
ros2 service call /pick_place/run std_srvs/srv/Trigger {}
```
**预期**：真实 UR5 缓慢走 预抓取→抓取→抬起→放置 轨迹（夹爪可先模拟）。
**判定**：轨迹安全、位置大致对准零件上方。
**常见问题**：
- 规划失败 → 目标不可达/超工作空间；检查 `place_pose`、抓取位姿是否在臂可达范围。
- TCP 不对 → 装夹爪后把 `tcp_link` 改成夹爪 TCP 连杆，并在 URDF 里加夹爪。

---

## 测试 9：整机闭环（放真零件）

1. 料框放几个零件，夹爪接真机（`simulate:false`）。
2. 跑测试8的总启动。
3. 触发 `/pick_place/run`，观察：识别→抓起→放到指定位置。
4. 多次触发，统计**抓取成功率**。

**判定（阶段验收）**：能从无序堆叠中稳定抓起零件并放到放置点。
**调优方向**：成功率低就回到测试5/6调感知与抓取标注；位置偏就查手眼标定精度。
